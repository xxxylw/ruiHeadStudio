# 方案 1 + 3：基于头部先验与课程采样的文本驱动 3D 头像训练路线

## 目标

这条路线的目标不是直接解决“像不像 Messi / Iron Man”的身份问题，而是先把当前系统里更基础、也更影响最终观感的两件事做稳：

1. 头部几何更稳定，尤其是侧脸、头部体积、轮廓和五官结构。
2. 表情与转头驱动更自然，减少嘴部、眼部和脖子在连续运动中的抖动、漂移和局部崩坏。

核心思路是把现有单阶段的 `text + SDS + ControlNet` 优化，改成“先学可驱动头部先验，再做文本外观定制”的两阶段 pipeline，并在第一阶段引入 `TalkVid` 的课程式采样与难例覆盖。

## 为什么要走这条路线

当前系统里，文本先验很强，但稳定的 3D 头部动态先验还不够强。直接让 `SDS / NFSD` 从零把“几何结构、可驱动性、外观语义”同时学出来，容易出现三个典型问题：

1. 正脸还能看，侧脸和大姿态容易崩。
2. 表情切换时局部结构不稳定，嘴、眼和下颌容易抖。
3. 文本优化会把几何往图像先验方向拉，导致连续驱动时不够稳。

`TalkVid` 的价值不在于“再多一点帧数”，而在于它提供了更多真实姿态、表情幅度、脖子转动和稀有动作边界。把它直接并进当前随机采样训练里，只能部分缓解分布问题；把它组织成先验训练和课程采样数据，才能更充分地发挥价值。

## 总体 Pipeline

整体流程拆成四层：

1. 数据组织层
2. 头部先验训练层
3. 课程采样与难例强化层
4. 文本驱动定制层

### 1. 数据组织层

输入数据不再只是“能被 dataloader 读到的 `.npy` / 目录”，而是要在进入训练前先形成结构化语料。

建议把 `TalkSHOW + TalkVid` 统一整理出以下元信息：

- `group`：`talkshow` / `talkvid`
- `source`：视频或 clip 级来源
- `sequence`：连续参数序列
- `frame`：单帧 FLAME 参数
- `difficulty bucket`：按姿态和表情幅度划分的难度桶
- `coverage tag`：稀有姿态、极端表情、快速转头等标签

其中 `difficulty bucket` 可以先做一个最小版本：

- `easy`：接近正脸，小表情，小幅 neck rotation
- `medium`：中等 yaw / pitch，中等表情幅度
- `hard`：大 yaw / pitch，大嘴张合，明显 neck rotation
- `rare`：数量少但重要的边界姿态和快速变化片段

这一步的目标是把 `TalkVid` 从“更多数据”变成“可调度的数据”。

最小落地版本里，可以先通过离线脚本 `scripts/build_pose_difficulty_metadata.py` 为每条序列生成难度元数据，再在训练时读取这些标签，而不是一开始就把难度判断塞进在线 dataloader。

### 2. 头部先验训练层

第一阶段不以文本 prompt 为核心，而是先学习一个更稳定的可驱动头部表示。

训练目标是：

- 给定 FLAME 条件，渲染出的头部结构保持稳定
- 多视角下几何一致
- 表情和姿态变化连续自然
- 相邻驱动状态之间不要出现明显抖动或结构漂移

这一阶段的核心不是让模型学“谁”，而是让模型先学会“一个头应该怎么稳定地动起来”。对当前系统而言，这相当于先把 `3DGS + FLAME` 的动态头部先验打牢，再把文本优化接回去。

第一阶段结束后，系统应当已经具备：

- 更稳的基础头部几何
- 更好的姿态覆盖能力
- 对连续驱动更友好的初始化状态

### 3. 课程采样与难例强化层

课程采样贯穿第一阶段，但也可以继续延续到第二阶段。

当前采样逻辑已经支持：

- `group -> source -> sequence -> frame`

下一步应当在其上增加“按训练阶段切换分布”的能力，形成：

- `stage -> difficulty bucket -> group -> source -> sequence -> frame`

一个务实的课程学习节奏可以是：

- 前期：以 `easy` 和一部分 `medium` 为主，先把正脸和小姿态学稳
- 中期：增加 `hard`，强化侧脸、大表情和大转头
- 后期：加入 `rare` 和高动态片段，补齐边界能力

在这个策略里，`TalkSHOW` 的作用更像提供较干净、较稳定的基础分布，`TalkVid` 则主要负责拓宽覆盖范围和补齐难姿态、难动作。

这样做有两个直接收益：

1. 训练初期不容易被极端姿态拖崩。
2. 后期能系统性覆盖之前容易失败的边界情况，而不是依赖纯随机命中。

### 4. 文本驱动定制层

在已有头部先验之上，再接回当前文本驱动部分：

- text prompt
- `SDS / NFSD`
- `ControlNet pose/depth`

这时文本优化的职责应当被收缩为：

- 决定外观语义和风格
- 把已有的稳定头部表示往目标角色或风格上推

而不再让它同时承担“从零造出稳定几何”和“学习头部驱动规律”的职责。

这会带来两个预期改进：

1. 文本优化更稳，不容易把几何拉坏。
2. 在相同 step 下，更容易得到可用的侧脸、轮廓和动态表现。

## 训练阶段建议

建议把训练拆成两个显式阶段：

### 阶段一：可驱动头部先验训练

输入：

- `TalkSHOW + TalkVid`
- FLAME 参数序列
- 课程采样策略

目标：

- 学稳定的动态头部先验
- 覆盖从简单到困难的姿态空间

产出：

- 一个可继续做文本定制的稳定初始化模型

### 阶段二：文本驱动外观定制

输入：

- 阶段一模型
- text prompt
- `ControlNet pose/depth`

目标：

- 保留阶段一已经获得的几何与动态稳定性
- 把外观拉向目标文本语义

产出：

- 具有较好驱动稳定性和更强文本一致性的 3D 头像

在当前工程里，这两个阶段更适合先落成两份独立配置文件，而不是做 YAML 继承。原因是现有 `load_config` 只负责读取和合并传入配置，不提供 `extends` 语义。

对应地，训练启动也可以先固定成两条脚本入口：

- `scripts/run_stage1_prior.sh`
- `scripts/run_stage2_text.sh`

这样可以把环境变量、GPU 选择和阶段配置固定下来，减少每次手工改命令时的漂移。

## 数据采样策略建议

如果只做最小可用版本，建议先实现三件事：

1. `TalkSHOW` 和 `TalkVid` 分组权重可配
2. `easy / medium / hard / rare` 四档难度桶
3. 不同训练阶段使用不同的桶采样权重

第一版不建议一开始就做太复杂的自适应策略，比如在线难例挖掘或复杂 curriculum scheduler。先把静态课程采样跑通，更容易验证收益。

一个简化示例：

- 阶段一前 30% step：
  - `easy: 0.6`
  - `medium: 0.3`
  - `hard: 0.1`
  - `rare: 0.0`
- 阶段一中 40% step：
  - `easy: 0.3`
  - `medium: 0.4`
  - `hard: 0.25`
  - `rare: 0.05`
- 阶段一后 30% step：
  - `easy: 0.1`
  - `medium: 0.3`
  - `hard: 0.4`
  - `rare: 0.2`

## 评估方式

这条路线不能只看最终生成图“顺不顺眼”，需要单独评估几何稳定性和驱动自然度。

建议至少看四类指标：

1. 姿态覆盖与边界表现
   - 大 yaw / pitch 样本是否明显更稳
   - 稀有姿态是否仍然崩坏

2. 连续驱动稳定性
   - 连续序列中嘴部、眼部、颈部是否抖动
   - 轮廓和头部体积是否在相邻帧间明显跳变

3. 文本外观一致性
   - prompt 驱动下角色风格是否更清晰
   - 不同角度下外观是否一致

4. 训练稳定性
   - 是否更容易收敛
   - 是否更少出现早期几何崩坏

## 这条路线解决什么，不解决什么

这条路线主要解决：

- 几何更稳
- 动作更自然
- 文本优化时不那么容易把几何拉坏

这条路线暂时不直接解决：

- “像 Messi 这种具体身份到底像不像”的问题

因为“像谁”更多依赖身份约束或参考图，而不是仅靠更大的驱动数据集。换句话说，`TalkVid` 更擅长帮系统学会“怎么更自然地动”，不直接负责“长得更像目标人物”。

## 推荐落地顺序

为了控制风险，建议按下面顺序推进：

1. 先补充数据元信息与难度桶，验证 `TalkVid` 的课程采样是否可跑通。
2. 再把训练拆成显式的“两阶段”流程，先验训练与文本定制分开。
3. 最后再决定是否引入身份约束分支，专门解决“像谁”的问题。

这个顺序的好处是，每一步都能单独验证收益，不会把所有复杂度一次性压到同一轮改造里。
